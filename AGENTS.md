# Repository Guidelines

## Project Structure & Module Organization
- `src/` holds the runtime implementation, grouped by subsystem (`app/`, `physics/`, `render/`, `input/`, `config/`, `command/`). Each module has a matching header in `include/`.
- `include/` mirrors `src/` and exposes public interfaces; add new modules as `include/<module>/...` + `src/<module>/...` pairs.
- `config/` contains runtime JSON and presets (`config/app.json`, `config/custom_preset.txt`, and `config/objects/*.asset.json`).
- `tests/` contains the C test harness (`tests/collider_test.c`).
- `build/` is generated by `make` and is safe to delete.
- `export/` and `import/` store frame output and legacy shape inputs for tooling.

## Build, Test, and Development Commands
- `make` builds the `physics_sim` binary in the repo root.
- `make run` builds and launches the SDL app locally.
- `make collider-tests` builds and runs the headless collider test harness.
- `make shape_asset_tool` or `make shape_mask_tool` builds CLI tools for converting or validating shape assets.
- `make clean` removes `build/` and the `physics_sim` binary.

## Coding Style & Naming Conventions
- C11 (`-std=c11`) with 4-space indentation and braces on the same line as the control statement.
- Functions, files, and variables use `snake_case` (e.g., `scene_controller_run`, `shape_asset_tool.c`).
- Keep module boundaries clear: public APIs in `include/`, private helpers in `src/`.

## Testing Guidelines
- Primary test is the collider harness in `tests/collider_test.c`.
- Run locally with `make collider-tests` before large physics changes.
- Add new C tests under `tests/` and wire them into the `makefile` if they need custom sources.

## Commit & Pull Request Guidelines
- Commit messages are short, Title Case summaries (examples in `git log`: “Colliders Perfected”).
- Keep commits focused; avoid mixing refactors with feature changes.

## Configuration & Assets
- Update `config/app.json` for defaults; avoid hardcoding values in code when a config entry exists.
- Place new shape assets in `config/objects/*.asset.json`; legacy shapes go in `import/` and should be converted with the CLI tools.
